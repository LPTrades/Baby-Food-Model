## Version Update - 2026/05/28
## Changes Added

* CISD + IFVG detection logic replacing the old FVG-only model
* Range definition: from the recent 5m swing high timestamp to the lowest low after the swing low break
* Consecutive down-close run detection inside that range, ending at the lowest low
* CISD level = open of the first candle in the run (highest open in the run)
* SL = lowest close in the run (body-based, no wicks)
* IFVG trigger = top of the most recent bearish FVG whose middle candle (C2) falls inside the consecutive run
* Long fires when 1m close > CISD AND > IFVG trigger simultaneously
* Short mirrors everything off swing high breaks
* Opposite-direction signal while a trade is active: existing trade closed at that bar's close (result='displaced'), new trade opens on the same bar
* Exception to above: if the new opposite trade would be skipped (wide stop >80pts), the existing trade is NOT displaced and keeps running
* Same-direction re-trigger while a trade is open: blocked entirely
* Configurable year filter via BACKTEST_YEARS constant (single year, list, range, or None for all data)
* Output filename reflects the year range used (e.g. backtest_trades_2020_2026.csv)
* Configurable RR target (prompt at runtime, default 1.0)
* Random seed input for reproducible 20-example trade selection
* Wide-stop skip filter: open_trade() returns None if natural SL distance exceeds max_stop (80pts); setup is consumed so it doesn't re-fire on subsequent bars; skip count tracked and shown in stats output separately from trade count

## Bugs Fixed

* Removed stop-capped logic (natural_sl / capped distinction) — backtester now either takes the trade at its natural stop or skips it entirely; no more partial/adjusted stops polluting the stats
* Wide-stop skip no longer displaces an opposite active trade — existing trade keeps running when a new setup is skipped

## Reason For Changes

* FVG-only model lost across the board. Moved to CISD + IFVG as the new entry model to test
* Wide-stop filter added to isolate whether high-volatility setups with blown-out stops are dragging down performance; skip count surfaces how often that's happening

## Observations

* If skip count from the 80pt filter comes back high, it signals the model is triggering most often during volatile expansion — worth logging as a separate insight
* BACKTEST_YEARS = [2026] is the default; change to None or a range for multi-year results
* Stats output shows: total trades, win rate, avg R, skipped (wide stop) count — check skip count on first run to calibrate whether 80pts is the right threshold
